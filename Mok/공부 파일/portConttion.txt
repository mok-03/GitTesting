Serial Option
{
 Baud Rate : 시리얼의 통신 속도 설정 , 단위 bit/sec (통신속도가 빠를수록 노이즈에 취약하거나 왜곡될 확율이 높음)						ㄴ 그럼으로 적당한 속도 설정이 필요함

  Data Bits  : 시리얼통신을 얼마의 비트로 통신을 할것인지(Base : 8bit

  Parity Bit  : 패리티 검사를 하는경우 옵션 으로 선택 가능, 일반적으로NONE을 하지만 통신의 정확도를 높이기 위해 설정하는 경우가 있음
		ㄴ정보 전달과정에서 오류가 생겼는지 검사하기위해 추가된 비트
		(Even Parity 와 Odd Parity)
		{
			이진수 통신에서 오류검출
			0001101 1 일때 Even Parity(짝수 페리티) 일땐
			1이 3개 임으로 1을 추가로 줘서 짝수를 만들어준다.

			이미 홀수 일경우 0
			
			만약 1001101 1 일경우 Even Parity는 에러가 났다는것을 알수있다
				ㄴ parity 까지 더했을때 1이 홀수다.
		}

   Stop Bits : 시리얼 통신을 할때 통신 말미에 지정 , 일반적으로 1 지정

 Flow Control : 흐름제어는 물리적인 비트를 별도로 두어 양방향으로 통신하는 경우 통신의 무결성을 높이기 위해 사용함, 일반적으로 사용하지않음

CR/LF
{
 타자기에서 내려온 이름
 CR : Carriage Return (\r)
 LF : Line Feed (\n)

 LF 커서위치를 그대로 두고 종이를 한라인 올리는 동작
 CR 현재 라인에서 커서의 위치를 맨앞으로 옮기는 동작
 window & DOS OS -> CRLF (CR + LF) 커서를 앞으로 옮기고 종이를 위로 옮기는 동작
 Unix & Linux & C OS -> LF
 (unix 에서 만든 문서에서 LF 형식을 사용하기 때문에 CRLF 에선 줄바꿈이 안됨)

}




데이터 전송 원리
{
송신측 호스트의 응용 프로그램에 보내는 데이터를 수신측 호스트의 응용 프로그램에 전송하려면 각 프로토콜에서 정의한 제어정보 가 필요합니다 (주소,포트번호 등)
 제어 정보는 위치에 따라 앞쪽에 붙는 헤더 (Geader)와 뒤쪽에 붙은 트레일러(trailer)로 나뉩니다. 이러한 제어 정보가 결합된
형태로 전송되며, 이를 패킷(packet)이라고 부릅니다.
}

TCP (Transmission Control Protocol)
{
	 상대방의 ip주소와 port 를 알고 있어야 연결요청을 할수있다.

	 3 way handshake 방식 서로 포트를 확인하고 연결하기 위해 3번 요청 응답후 연결됨
  		ㄴ( client ->(syn) ->server ->(ACK+SYN)->client ->(ACK)->server)
  	 	ㄴ이 과정에서 많은 시간이 소요된다고 함

	TCP는 서로 계속 연결되어 있어서 보낸 데이터가 제데로 넘갔는지도 확인이 가능함

	TCP 함수들
	{
	socket(int domain, int type,int protocol)
		ㄴ 소켓을 만드는데 이 함수를 사용함 소켓 역시 파일로 다루어지기 떄문에 반환값이 파일 디스크립터입니다. 실패시 -1리턴
			

	connect(int fd, struct sockaddr *remote_host,socklen_t addr_length)
		ㄴ 원격 호스트(원격 컴퓨터)와 연결하는 함수 입니다. 연결된 정보는 remote_host에 저장됩니다. 성공시 0 오류시 -1리턴
			(블럭 방식으로 동작한다, 연결 요청에 대한 결과가 결정 되기전에는 끝나지 않는다. 실행되자말자 바로 리턴될것이 아니다.
	
	bind(int fd, struct sockaddr *local_addr,socklen_t addr_length)
		ㄴ 소켓을 바인딩 한다. fd로 넘겨지는 소켓과 이 프로세스와 묶는다, 그래서 해당 프로세스 소켓을 통해 다른컴퓨터로부터 연결을 받아드린다.
			ㄴ ip주소와 포트 번호를 결합 

	listen(int fd,int backlog_queue_size)
		ㄴ 소켓을 통해 들어오는 연결을 듣습니다. backlog_aueue_size 만큼
			ㄴ 요청에 대한정보는 알수없고 return 되는 값은 클라이언트 요청이 수신되는 경우, 에러 발생

	accept(int fd,sockaddr *remote_host,socklen_t *addr_lenght)
		ㄴ 어떤 컴퓨터에서 이컴퓨터로 연결을 할때 연결을 받아 들입니다.
		ㄴ연결된 원격 컴퓨터의 정보는 remote_host에 저장됩니다. 오류시 -1리턴
			

	-------------연결 

	 send(int fd, void* buffer, size_t n, int flags) 데이터 전송
		ㄴbuffer를 소켓 파일 디스크립터인 fd로 전송합니다. 보낸 바이트수를 반환하며 실패시 -1을 리턴 (제대로 넘어갔는지 확인)

	  recv(int fd, void* buffer, size_t n, int flags) 데이터 수신
		ㄴsend함수와 사용법이 거의 비슷합니다. n바이트를 buffer로 읽습니다. 성공시 받은 바이트수를 반환하며 실패시 -1을 리턴.
	
	  close()
	    ㄴ 최초 api를 통해 생성한 서버 소켓 + accpet 호풀에 의해 생성된 소켓도 관리해줘야 한다.

	}

요청을 받아들이는 곳에서는 어떤 연결 요청(일반적으로 포트 번호로 식별)을 받아들일 것인지를 
미리 시스템에 등록하여, 요청이 수신되었을 때 해당 요청을 처리할 수 있도록 준비해야 합니다.
요청을 받아들이는지에 따라 소켓의 역할이 나뉘게 되는데, 전자에 사용되는 소켓을 클라이언트
 소켓(Client Socket), 후자에 사용되는 소켓을 서버 소켓(Server Socket)이라고 합니다.	
그런데 여기서, 오해하기 쉬운 내용이 있습니다. 앞서 설명한 내용을 보면 마치 클라이언트 소켓
(Client Socket)과 서버 소켓(Server Socket)이 태생적으로 구조가 다른, 전혀 별개의 소켓(Socket)인 것처럼 여겨질 수 있다는 것이죠.
 하지만 두 소켓(Socket)은 동일합니다. 소켓의 역할과 구현 절차 구분을 위해 다르게 부르는 것일 뿐, 전혀 다른 형태의 소켓이 아니라는 것이죠. 
단지 역할에 따라 처리되는 흐름, 즉, 호출되는 API 함수의 종류와 순서들이 다를 뿐입니다.
	
}

시리얼과 tcp 는 앞에 패킷의 크기를 보내야 한다 
 ㄴ이유는 tcp나 시리얼은 너무길면 짤라서 보내기 떄문에 한번에 온다고 보장할수 없다.
    그렇기 때문에 패킷의 크기를 맨앞에 배치시켜 받은 데이터의 크기를 받은뒤 비교후
    맞지않으면 다음 패킷을 받아 채워넣는다.

udp의 경우 메시지향 이기떄문에
보내는 길이만큼 데이터가 오기때문에 자를필요가 없다고 한다.


TcpClient로 직접 소켓을 바인드해서 '비동기'로 리슨과 커넥을 해보았으니, SocketAsyncArgs를 활용해 (이것 또한 비동기 형식) 프로그램을 짜보는 것도 추천합니다.  몰론 이젠 알고 계실지는 모르겠지만요.  또한 게임 클라이언트끼리 연결하는 Peer To Peer 방식은 대부분 UDP로 제작합니다. 그렇다고 TCP/IP로 못하는 건 아닌데, 만일 TCP로 PeerToPeer (P2P)를 구현 하려면 그 해당 Peer이 포트가 포워딩이 되어 있는 상태에서 진행해야 합니다. 말그대로 게임을 이용하시는 사용자(고객)은(는) 포트포워딩을 모르는게 다반사 일것이니 UPnp 포트 맵핑을 해줘야 하는건 덤이지요... 심지어 어떤 공유기는 UPnp 포트 맵핑이 거부되는 공유기도 존재합니다. 한마디로 NAT에게 UPnp통신을 바랄 수 밖에 없게 되죠 반면 UDP의 경우 홀 펀칭 일명 Relay(STUN)라는 NAT 우회 기법이 존재하므로 한번 UDP를 통해서 Peer To Peer도 공부해 보시면 대략 게임 서버 뿐만이 아닌 다른 서버들의 흐름도 알게 됩니다.
(아. 그렇다고 홀펀칭이 무조건 성공한다는 보장도 없습니다. 하지만 통계에 따르면 80%확률 이상의 성공률을 보이고 있습니다.)

PeerToPeer을 공부하시는걸 권장하는 이유는 대부분의 서버는 1초당 4번 동기화 작업을 진행해야 하는데 액션을 추구하는 던파, 서든, 카스등 총게임/액알(액션 알피지)는 대부분 Peer To Peer을 통해 UDP 로 비연결식 통신을 하여 속도를 취하기 떄문에 액션을 추구하는 게임을 제작하고 싶다면 꼭 구현해야 하는 존재이지요, 이렇게 P2P는 현대에서 필수적은 기술로 자리잡고 있기 때문에 한번 권장해 봅니다.
